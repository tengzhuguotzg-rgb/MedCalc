## 逐条回应

### 1. 截断字符数：7000→5000？

对方认为7000字符/token换算偏乐观，建议降到5000。

**我的实测数据**：
- 指南中位数2253字符，91%在5000字符以下 → 截断对指南几乎无影响
- 教材中位数19680字符，8章超5万 → **截断才是真正的问题**

关键问题不是"截断多少"，而是**截断对教材意味着什么**：
- 教材单章2万-20万字符
- 截断到5000或7000都是只取前1/4~1/3
- 教材结构是：概述→病理生理→诊断→治疗→并发症→延伸
- 前5000字符可能只到病理生理部分，治疗建议全丢了

**所以我不同意简单降数字。截断策略必须区分体裁**：
- 指南：大部分<5000字符，截断几乎无感，上限8000足够
- 教材：截断是不可避免的（2万-20万字符），但不应只取前N字符
- **教材应该智能截取**：优先取与diag.name相关的段落，而非前N字符

**修正方案**：
```
指南: 不截断（99%<10k字符，不超预算）
教材: 截取策略——
  方案A（简单）: 取前N字符，N按预算分配
  方案B（更好）: 取标题+第一段(概述)+包含查询关键词的段落±上下文
```

方案B更好但实现复杂，v3先用方案A，后续可优化。

**token预算重算**（基于实测）：

```
场景: 典型重症肺炎
验证通过: 8篇指南(~2500字均) + 3章教材(~20000字均)

指南: 8 × 2500 = 20,000字符 ≈ 13,000 tokens
教材: 3 × 教材截断 = ?
预算剩余: 58000 - 13000 = 45,000 tokens ≈ 67,500字符
教材每篇可分配: 67,500 / 3 = 22,500字符

所以3章教材各取22,500字符完全够，远超7000/5000
```

但最坏情况——8章教材：
```
指南: 0篇
教材: 8 × 截断 = ?
预算: 58000 tokens ≈ 87,000字符
教材每篇: 87000 / 8 = 10,875字符
```

**结论：截断不是固定数字，是按预算和篇数动态分配的**。
公式: `perPiece = min(文章总字符数, totalBudget / totalPieces)`
totalBudget = 87000字符（基于58000 tokens × 1.5字符/token）

**对方建议5000是保守但粗糙的。动态分配更精确且不会浪费预算。**

### 2. 全部拒绝时回退量从10→5

**我不同意。**

理由：
- 对方说"LLM全部拒绝很可能真的没相关文档"——这是对的
- 但对方忽略了：**回退结果会经过提取agent的段落级过滤**
- 提取agent的prompt明确要求"只提取与诊断主题直接相关的内容，排除不相关段落"
- 5条噪声文章喂入提取agent → 提取agent过滤后可能0条输出 → summary为空
- 10条噪声文章喂入 → 提取agent可能从中找到1-2条勉强相关的 → 至少有内容
- **在医疗场景，空结果比噪声更危险**——医生看到空会以为系统不支持这个诊断，看到有少量内容至少知道有参考可看

但我承认回退10条全是噪声时，提取agent可能硬凑出错误建议。

**折中方案**：回退10条，但在UI上标注"⚠️ 以下建议由低相关性文献提取，请谨慎参考"。这样既不丢失潜在有用的内容，又提醒医生谨慎。

### 3. 体裁判断交给LLM

**我同意。**

v3已经设计了type字段（"推荐意见"|"知识要点"），让LLM自己判断体裁比硬编码规则更灵活。prompt里给几个示例即可：

```
判断体裁的依据：
- 如果文章包含"我们推荐"、"建议"、"Grade 1A"等明确推荐句式 → "推荐意见"
- 如果文章是叙述性知识（机制阐述、病理描述、综述）→ "知识要点"  
- 如果两者都有 → "混合"，每条分别标注type
```

### 4. LLM调用失败回退

**我同意，这是必须加的。**

每一步LLM调用都应有try-catch + 回退：

| 步骤 | LLM调用 | 失败回退 |
|------|---------|---------|
| Step 1 查询扩展 | expandQueryWithLlm | 不扩展，用原始diag.name搜FTS |
| Step 3 语义验证 | verifyResultsWithLlm | 不验证，直接用FTS排序取前10 |
| Step 5 提取归纳 | extractAndSummarize | recommendations=[], summary="AI服务暂不可用" |

这是鲁棒性基本要求，v3遗漏了，必须补。

### 5. 空文档检测

**我同意，但这是小概率事件。**

kb_content表是buildIndex时写入的，如果写入成功content不可能为空。但加载时加个检查成本为零，应该加。

---

### 6. 医生反馈字段

**我不同意现在加。**

理由：
- 当前核心问题是搜索质量和提取质量，不是反馈闭环
- feedbackRating字段看似简单，但后续利用它影响检索策略需要大量额外逻辑
- 过早加反馈机制可能收集到的是"UI不好看"而非"建议不准"的噪声
- **这是第二阶段的事**，先把v3的检索→验证→提取链路跑通，确认质量OK后再加反馈

---

### 7. 提取agent输出"哪些文章用了/没用+原因"

**我不同意现在加。**

理由：
- 增加输出复杂度和token消耗
- LLM解释"为什么没用某篇文章"本身就是低质量信息（LLM倾向编造合理化解释）
- 透明性可以通过更简单的方式实现：UI显示"参考了X篇文献"即可
- **医疗场景需要简洁可操作的建议，不是LLM的自我解释**

---

### 关于FTS 50条性能

对方建议实测。**同意，实施后必须实测。** 但481篇KB的FTS索引量很小（远小于万级），50条vs30条不太可能有可感知差异。如果真慢，加LIMIT即可。
