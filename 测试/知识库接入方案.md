# 知识库接入方案

## 一、核心理念

**知识库不是被动参考书，是主动驱动源。**

```
❌ 旧思路：诊断→贴指南链接→用户自己看
✅ 新思路：诊断→读知识库→LLM提取建议→直接展示具体推荐意见
```

知识库的三个层次用法：

| 层次 | 做什么 | 知识库角色 | 对应课题组件 |
|:---|:---|:---|:---|
| L1 溯源 | 诊断→相关指南链接 | 被动参考书 | 基础功能 |
| L2 建议 | 诊断→指南原文→LLM提取具体推荐→展示 | **主动驱动** | 检索Agent+推理Agent简化版 |
| L3 决策 | 诊断→指南工作流→逐步引导医生操作 | **决策定义** | 完整多Agent架构（后期） |

本次实现L1+L2，L3作为后续扩展。

## 二、架构

```
resources/rawfile/kb/          (431页md，已就位)
         ↓ 首次启动遍历
KnowledgeService               (FTS5索引 + 搜索 + 读取)
         ↓ search(query) → SearchResult[]
         ↓ getPage(path) → KnowledgePage
    ┌────┼────┐
    ↓    ↓    ↓
 诊断详情  计算器结果  独立搜索
    ↓
 LLM提取指南建议
    ↓
 直接展示具体推荐意见
```

**完整数据流（以脓毒性休克为例）：**

```
① 诊断引擎命中 septic_shock（确定性，零幻觉）
        ↓
② KnowledgeService.search("感染性休克 脓毒症 乳酸")
   → 返回3个相关wiki页
        ↓
③ KnowledgeService.getPage() 读取top3页面内容
        ↓
④ 构建LLM请求：
   system: "从以下指南原文中提取与当前诊断相关的推荐意见"
   context: 3页wiki原文（约10-15KB）
   patient: SOFA=14, MAP=52, 去甲肾上腺素
   diagnosis: 感染性休克
        ↓
⑤ LLM返回结构化建议（只从给定文本提取，不自行推理）
        ↓
⑥ 展示：推荐意见 + 证据级别 + 来源 + 查看原文入口
```

**与课题架构的对应关系：**

```
课题完整架构：调度Agent → 检索Agent → 推理Agent → 计算Agent
App简化版：  诊断引擎(计算) → KnowledgeService(检索) → LLM单轮提取(推理)
```

简化版保留了课题的核心创新——确定性引擎做计算、知识库驱动推理、LLM只做自然语言接口——但不需要多Agent通信协议，实现成本低得多。

## 三、KnowledgeService职责

| 方法 | 功能 | 说明 |
|:---|:---|:---|
| `initIndex()` | 首次启动建FTS5表 | 遍历rawfile/kb/所有md，读内容插表，建完标记已完成 |
| `search(query, limit?)` | 全文检索 | FTS5 MATCH查询，返回`SearchResult[]`（路径+标题+片段+相关度） |
| `getPage(path)` | 读取单页内容 | 异步读rawfile，返回完整md文本，供阅读页渲染或喂给LLM |
| `searchAndLoad(query, limit?)` | 搜索+加载 | 先search取topN路径，再getPage批量读内容，返回`KnowledgePage[]` |

## 四、数据结构

```typescript
class SearchResult {
  path: string = '';          // '脓毒症/SCCM/SCCM-SSC-3-血流动力学管理.md'
  title: string = '';         // 从YAML frontmatter或第一行H1提取
  snippet: string = '';       // 匹配关键词前后各100字
  rank: number = 0;           // FTS5 bm25相关度
}

class KnowledgePage {
  path: string = '';
  title: string = '';
  content: string = '';       // 完整md内容
  sourceInfo: string = '';    // 从frontmatter提取的来源信息（指南名+年份+机构）
}

class GuidelineRecommendation {
  text: string = '';          // "去甲肾上腺素为一线血管活性药"
  evidenceLevel: string = ''; // "强推荐" / "条件推荐" / "Good Practice Statement"
  source: string = '';        // "SSC 2026 Rec 44"
  sourcePage: string = '';    // wiki页面路径，点击查看原文
}

class DiagnosisAdvice {
  diagnosisName: string = '';   // "感染性休克"
  patientContext: string = '';  // "SOFA=14, MAP=52mmHg, 去甲肾上腺素使用中"
  recommendations: GuidelineRecommendation[] = [];
  sourcePages: string[] = [];   // 所有引用的wiki页面路径
}
```

## 五、FTS5索引策略

```
CREATE VIRTUAL TABLE kb_index USING fts5(
  title,          -- 页面标题（权重高）
  content,        -- 页面全文（权重低）
  path UNINDEXED, -- 文件路径，不参与搜索
  source UNINDEXED,
  tokenize='unicode61'  -- 支持中文分词（简易，按字拆分）
)
```

- title权重高于content（标题匹配排前面）
- unicode61按字拆分，中文够用（"脓毒症"→"脓""毒""症"三token，都能匹配）
- 首次启动建索引约2-3秒，存RDB，后续启动不重复建
- index目录浏览作为独立搜索的分类导航，与FTS5互补

## 六、LLM提取指南建议

### 核心原则

**LLM只做提取和整理，不做推理和决策。** 它的任务是：从给定的指南原文中，找出与当前诊断相关的推荐意见，用结构化格式返回。答案必须在给定的文本中有依据，不允许自行编造。

### Prompt模板

```
你是ICU临床指南助手。以下是当前患者情况和从指南原文中检索到的相关内容。

## 患者情况
- 诊断：{diagnosisName}
- 关键数据：{patientContext}

## 指南原文
{page1_content}
---
{page2_content}
---
{page3_content}

## 任务
从以上指南原文中提取与该诊断直接相关的临床推荐意见。要求：
1. 只提取原文中明确存在的推荐意见，不要推断或编造
2. 每条推荐意见注明证据级别（如"强推荐""条件推荐"等，原文有就标，没有就不标）
3. 每条注明来源（指南名+推荐编号，如"SSC 2026 Rec 44"）
4. 按临床优先级排序（强推荐在前，条件推荐在后）
5. 最多提取8条最相关的

## 输出格式（JSON）
{
  "recommendations": [
    {
      "text": "推荐意见原文或概括",
      "evidenceLevel": "强推荐/条件推荐/Good Practice Statement/未标注",
      "source": "来源标注"
    }
  ]
}
```

### 幻觉控制

| 机制 | 说明 |
|:---|:---|
| 上下文限定 | LLM只能看到3页wiki原文，不访问外部知识 |
| 任务限定 | "只提取原文中明确存在的"，不是"请给出治疗建议" |
| 可溯源 | 每条推荐意见标注来源页+推荐编号，用户可点进原文核实 |
| LLM确认复用 | 复用现有LLM诊断确认机制，建议也可标记confirmed/rejected |

### 上下文大小估算

- 3页wiki × 3-5KB/页 = 9-15KB ≈ 4000-6000 tokens
- 加上prompt和患者数据 ≈ 7000 tokens总输入
- Qwen2.5-7B或DeepSeek-Chat的上下文窗口足够（32K+）
- 输出约500-1000 tokens

## 七、三个接入点

### 1. 诊断→指南建议（优先做，L2级）

触发时机：诊断详情页展示时

query构造：
```typescript
const query = match.rule.name;  // "感染性休克"
// 结果少于3条时追加
const extraQuery = match.rule.description;  // "脓毒症+血管活性药+乳酸>2mmol/L"
```

### 2. 计算器→指南参考（L1级，只贴链接）

触发时机：计算结果展示时

query构造：
```typescript
const query = calc.name + ' ' + calc.group;  // "SOFA评分 感染"
```

展示：计算器结果卡片底部加"📖 指南参考"链接列表，点击进入阅读页。暂不做LLM提取（计算器结果本身已有分级解读，不需要额外建议）。

### 3. 独立搜索（L1级，浏览+搜索双模式）

- 顶部搜索框 → FTS5全文检索 → 结果列表
- 下方分类导航 → 复用wiki/index.md目录结构 → 点击进入专题
- 搜索结果点击 → 进入阅读页

## 八、知识阅读页

新建`KnowledgeReaderPage`，Markdown渲染：

- 顶部：标题+来源信息（从frontmatter提取）
- 中部：Markdown内容渲染（ArkUI `RichText`或Web组件）
- 底部：wikilink解析，显示"相关页面"列表可点击跳转

wikilink格式如 `[[脓毒症/SCCM/SCCM-SSC-0-概述]]`，点击后调`getPage()`加载新页面。

## 九、实施步骤

| 步骤 | 内容 | 工期 | 依赖 | 知识库层次 |
|:---:|:---|:---:|:---:|:---:|
| 1 | 新建`KnowledgeService.ets` — initIndex/search/getPage/searchAndLoad | 2天 | 无 | 基础 |
| 2 | 应用启动时调用initIndex | 0.5天 | 步骤1 | 基础 |
| 3 | 新建`KnowledgeReaderPage.ets` — Markdown渲染页 | 1.5天 | 步骤1 | L1 |
| 4 | 诊断详情页L1接入 — 搜索+展示相关指南链接 | 1天 | 步骤1+3 | L1 |
| 5 | 诊断详情页L2接入 — LLM提取指南建议+展示 | 2天 | 步骤4 | **L2** |
| 6 | 计算器结果接入 — 搜索+展示指南参考链接 | 0.5天 | 步骤1+3 | L1 |
| 7 | 独立搜索入口 — 分类导航+全文搜索 | 1天 | 步骤1+3 | L1 |
| 8 | wikilink跳转 — 阅读页内点击跳转 | 0.5天 | 步骤3 | L1 |

**核心闭环：步骤1-5（7天），做完即可演示"诊断→指南建议→原文溯源"完整链路。**

步骤5是关键——它是知识库从"参考书"升级为"驱动源"的转折点，也是课题"确定性引擎+知识库驱动+LLM接口"架构在App侧的最小可行验证。

## 十、与课题架构的映射

| 课题组件 | 课题描述 | App简化版实现 | 对应步骤 |
|:---|:---|:---|:---:|
| 计算Agent | 确定性评分引擎 | 诊断引擎 + 41个计算器（已完成） | — |
| 检索Agent | 知识库检索 | KnowledgeService.search | 步骤1 |
| 推理Agent | 临床推理+建议 | LLM单轮提取（prompt限定） | 步骤5 |
| 调度Agent | 意图识别+任务路由 | 诊断引擎命中自动触发 | 步骤4 |

App简化版不需要Agent间通信协议、不需要上下文隔离（因为是单轮提取，不是多轮对话），但**核心架构一致**——课题论文可以写"移动端实现了课题核心架构的轻量级验证"。
