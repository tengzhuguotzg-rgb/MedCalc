# 诊断→建议 全流程重设计方案

## 设计原则

1. **搜索只做一次**：阶段1搜索的结果直接复用给阶段2，不重复搜索
2. **LLM调用最小化**：每步有明确不可省略的理由，不做冗余调用
3. **相关性逐层收紧**：FTS广搜(松) → 文章级验证(中) → 段落级提取(紧)
4. **懒加载**：用户不点进详情页，就不做昂贵的提取/归纳

## 完整流程

```
患者数据
  │
  ▼
┌─────────────────────────────────────┐
│ Step 0: 诊断触发（0次LLM）          │
│ ├ 硬规则匹配 → rule诊断             │
│ ├ 无硬规则命中 → LLM兜底建议        │
│ └ 用户手动添加                      │
│ 产出: diag.name + diag.description  │
└────────────────────────────────────┘
  │
  ▼ 诊断卡片出现时触发
┌────────────────────────────────────┐
│ Step 1: 查询构造（0次LLM）          │
│ query = diag.name                   │
│ （只用diag.name，不拼description）  │
│ 原因：description词太碎，           │
│  拼入后FTS噪声反而增大              │
└────────────────────────────────────┘
  │
  ▼
┌────────────────────────────────────┐
│ Step 2: 查询扩展（LLM调用#1）       │
│ expandQueryWithLlm                  │
│ "重症肺炎" → CAP HAP VAP 呼吸机相关 │
│ 产出: expandedTerms[]               │
└────────────────────────────────────┘
  │
  ▼
┌────────────────────────────────────┐
│ Step 3: FTS广搜（0次LLM）           │
│ query + expandedTerms OR搜索        │
│ 取 limit×3 条（约30条）             │
│ 同时搜指南+教材（不分两次）         │
│ 产出: broadResults[]                │
└────────────────────────────────────┘
  │
  ▼
┌────────────────────────────────────┐
│ Step 4: 核心词过滤（0次LLM）        │
│ filterByCoreTerms                   │
│ 必须包含diag.name中的核心病名       │
│ 收紧到 limit×2 条（约20条）         │
│ 产出: filteredResults[]             │
└────────────────────────────────────┘
  │
  ▼
┌────────────────────────────────────┐
│ Step 5: 文章级验证（LLM调用#2）     │
│ verifyResultsWithLlm                │
│ 判断每篇文章主题是否关于该诊断      │
│ ≤3条时跳过验证（信任FTS+coreTerm）  │
│ 产出: verifiedResults[]（≤10条）    │
│                                     │
│ ★ 同时存入 kbData.searchResults     │
│ ★ 同时存入 kbData.verifiedPaths[]   │
│   （验证通过的path列表，供Step7用） │
└────────────────────────────────────┘
  │
  ▼ 诊断卡片展示指南参考列表
  │ （用户看到参考条目，可能点击进入详情）
  │
  ▼ 用户点击"查看建议"时触发
┌────────────────────────────────────┐
│ Step 6: 加载已验证文章全文（0次LLM）│
│ 从kb_content表按verifiedPaths加载   │
│ 每篇截断3000字符（从4000缩减）      │
│ 指南≤5篇 + 教材≤3篇                │
│ 产出: pages[]（全文内容）           │
└────────────────────────────────────┘
  │
  ▼
┌────────────────────────────────────┐
│ Step 7: 提取+归纳（LLM调用#3）      │
│ ★ 合二为一，一次调用同时完成：      │
│                                     │
│ 输入: 诊断名 + 患者摘要 + 全文      │
│ 指令:                               │
│  1. 只提取与{diag.name}主题直接     │
│     相关的推荐意见                   │
│  2. 忽略文中仅提及但非主题的段落    │
│     （如肺炎文章中的镇静方案、      │
│      谵妄评估、营养支持等）         │
│  3. 分4类：病理生理/诊断/监测/治疗  │
│  4. 每类按临床优先级排序            │
│  5. 同时生成200字临床要点摘要       │
│                                     │
│ 输出JSON:                           │
│ {                                   │
│   "recommendations": [...],         │
│   "summary": "临床要点摘要"         │
│ }                                   │
│                                     │
│ 产出: recommendations[] + summary   │
└────────────────────────────────────┘
  │
  ▼ 详情页展示完成
```

## LLM调用统计

| 调用# | 时机 | Agent | 作用 | 不可省略理由 |
|--------|------|-------|------|-------------|
| #1 | 诊断卡片出现 | expandQueryWithLlm | 查询扩展 | 医学术语同义扩展，FTS无法自行完成 |
| #2 | 诊断卡片出现 | verifyResultsWithLlm | 文章级验证 | 医学科普交叉引用多，FTS+关键词无法区分"关于"vs"提及" |
| #3 | 用户点进详情 | 提取+归纳agent | 段落级提取+摘要 | 从全文提取4分类建议需临床判断，必须LLM；归纳与提取合一次调用省token |

**总计：3次LLM调用**

对比现状：
- 现状：expand×1 + verify×1（阶段2） + expand×1 + verify×1（阶段3，smartSearch内部） + 提取×1 + 归纳×1 = **6次LLM调用**
- 优化后：**3次LLM调用**，减少50%

## 关键改动点

### 1. 搜索只做一次，结果复用

```
现状:
  loadDiagnosisKb → smartSearch(query1) → searchResults  ← 展示用
  loadDiagnosisAdvice → searchAndLoad(query2) → 全文      ← 提取用
                      → searchByPrefix(query3) → 教材全文  ← 提取用

改后:
  loadDiagnosisKb → smartSearch(diag.name) → verifiedResults ← 展示用
                                                        ← 同时存verifiedPaths
  loadDiagnosisAdvice → 用verifiedPaths加载全文           ← 提取用（不搜索）
```

### 2. query统一用diag.name，不拼description

现状拼入description分词的问题：
- "ARDS 重度" + "Berlin P/F<100 PEEP≥5" → FTS搜"P/F"匹配到所有含血气的文章
- "脓毒症" + "SOFA≥2 血管活性药" → "血管活性药"匹配到休克、心衰等所有用升压药的文章

改后只用 `diag.name`，由扩展agent负责补充同义术语。

### 3. 教材搜索合并进smartSearch

现状 `searchByPrefix` 独立调一次 `smartSearch`，浪费一次expand+verify。

改法：smartSearch的FTS广搜本身就跨所有source（指南+教材），验证通过后按path前缀分组取top-N即可。

### 4. 提取+归纳合一次LLM调用

现状分两步：
1. 提取agent → recommendations[]
2. 归纳agent → summary

合并为一次调用，输出JSON同时含recommendations和summary，省一次LLM调用和token。

### 5. 提取agent增加段落级相关性约束

现状提取agent的prompt只说"提取相关推荐意见"，没有强调排除跨主题段落。

改后prompt核心指令：
```
只提取与"{diag.name}"主题直接相关的推荐意见。
严格排除文中仅提及但非该疾病核心内容的段落，例如：
- 肺炎文章中的镇静镇痛方案 → 排除
- 肺炎文章中的谵妄评估 → 排除  
- 肺炎文章中的营养支持 → 排除
- 肺炎文章中的抗生素方案 → 保留
- 肺炎文章中的呼吸支持策略 → 保留
判断标准：如果去掉该段落，对{diag.name}的临床决策有实质影响吗？
如果没有，排除。
```

### 6. 全文截断从4000→3000字符

理由：
- 医学文章核心内容（定义、诊断、治疗）通常在前2000字符
- 后半部分常是鉴别诊断、延伸讨论、相关疾病提及
- 截断到3000字符减少跨主题段落喂入概率
- 8篇×3000字符=24000字符输入token，仍可控

## DiagnosisKbData结构调整

```
class DiagnosisKbData {
  ruleId: string = '';
  searchResults: SearchResult[] = [];     // Step5产出，卡片展示用
  verifiedPaths: string[] = [];           // Step5产出，Step6加载用
  recommendations: GuidelineRecommendation[] = [];  // Step7产出
  summary: string = '';                   // Step7产出
  loadingKb: boolean = false;            // Step1-5进行中
  loadingAdvice: boolean = false;        // Step6-7进行中
  verifyInfo: string = '';               // 调试信息
}
```

删除：`expanded`, `refsExpanded`, `loadingSummary`（归纳合并进提取）

## 实施顺序

1. KnowledgeService：`searchByPrefix` 改为从已验证结果中按path前缀筛选，不再独立搜索
2. KnowledgeService：`searchAndLoad` 改为从已验证paths加载全文，不再搜索
3. PatientDetailPage：`loadDiagnosisKb` 存 `verifiedPaths`
4. PatientDetailPage：`loadDiagnosisAdvice` 改用 `verifiedPaths` 加载全文
5. PatientDetailPage：合并提取+归纳为一次LLM调用
6. 提取agent prompt 增加段落级相关性约束
7. 全文截断 4000→3000
8. 删除 `loadDiagnosisSummary`（合并进Step7）
9. 清理冗余字段和代码
