# 诊断→建议 全流程重设计 v2

## 对v1的反思与修正

### 问题1：filterByCoreTerms不适用于非标准诊断名

v1保留了filterByCoreTerms（Step 4），用diag.name中的词做字符串包含匹配。
- "重症肺炎" → coreTerms=["重症","肺炎"] → "重症"是停用词被剥离 → 只剩"肺炎"
- 用户自定义诊断如"心源性休克代偿期" → coreTerms=["心源性","休克","代偿期"] → 匹配不灵活
- LLM suggestion诊断名完全不可控

**修正：删掉filterByCoreTerms。既然已有LLM验证agent，关键词过滤是冗余的，且不灵活。**

### 问题2：limit×3的含义不清晰

v1写"FTS广搜取limit×3条"。"limit"是最终需要的结果数（如10条），limit×3=30条是广搜时多抓的buffer，给后续验证agent留筛选空间。

**修正：不用limit×3这种表达式，直接写具体数字。**

### 问题3：教材搜索几乎无命中

原因已定位：
- 教材md文件**没有summary字段**
- FTS索引的meta = summary + pathKeywords + linkTexts
- 教材的meta只有"实用重症医学" + 章节名（如"机械通气技术"）
- 搜"重症肺炎"时，教材meta里不含"肺炎"，FTS根本匹配不到

**修正：给教材文件补充summary，或在索引时从content前200字符提取摘要填入meta。**

### 问题4：指南类型差异未考虑

知识库指南分两种：
- **结构化指南**（如SSC、KDIGO）：有明确推荐意见、推荐等级（Grade 1A/2B等），适合直接提取
- **自然段落式指南**（如NICE抗生素路径、病理生理机制文章）：无明确"我们推荐…"句式，是知识性叙述

现状提取agent的prompt要求提取"推荐意见+证据级别"，对自然段落式文章会：
1. LLM硬编造推荐等级（标注"未标注"）
2. 把叙述性知识强行包装成"推荐意见"

**修正：提取agent应区分两种体裁，自然段落式提取为"知识要点"而非"推荐意见"。**

### 问题5：全文截断依据不明确

现状：每篇文章喂入LLM时截断到4000字符。
问题：什么时候会截断？——**8篇文章×4000字符=32000字符输入，约8000 token，接近模型上下文窗口边界。**

截断的依据应该是总输入token预算，而非单篇字符数。应该按token预算分配每篇的字符额度。

```

  ▼

│ ★ diag.name是后续所有步骤的核心锚点       │

  │

│ Step 1: 查询扩展               （1次LLM） │
│                                           │

│                                           │

│ 取数: 最多30条（不分指南/教材，一起搜）   │
│                                           │
│ FTS匹配的是meta字段(title+tags+summary+   │

  │
  ▼
┌─────────────────────────────────────────┐
│ Step 3: 语义验证               （1次LLM） │

│    - "推荐意见": 有明确推荐等级的指南条目 │

│    统一归入4临床类别:                      │
│    病理生理/诊断/监测/治疗                 │
│ 4. 同时生成200字临床要点摘要              │
│                                           │
│ 输出JSON:                                 │
│ {                                         │
│   "recommendations": [{                   │
│     "text": "...",                        │
│     "evidenceLevel": "1A"|"2B"|"未标注",  │
│     "source": "SSC 2021|教材12章",        │
│     "category": "病理生理|诊断|监测|治疗", │
│     "type": "推荐意见|知识要点"            │
│   }],                                     │
│   "summary": "临床要点200字"              │
│ }                                         │
│                                           │
│ 产出: recommendations[] + summary         │
└─────────────────────────────────────────┘
```

## LLM调用统计

| 调用# | 时机 | Agent | 作用 |
|--------|------|-------|------|
| 0 | 诊断触发（条件性） | suggestDiagnosesWithLlm | LLM兜底诊断（仅硬规则未命中时） |
| 1 | 诊断卡片出现 | expandQueryWithLlm | 查询扩展 |
| 2 | 诊断卡片出现 | verifyResultsWithLlm | 语义验证 |
| 3 | 用户点进详情 | 提取+归纳agent | 段落级提取+摘要 |

**总计：3次LLM调用（不含条件性的#0）**

对比现状6次→3次，减少50%。

## 关键改动清单

### 1. 删掉filterByCoreTerms + extractCoreTerms
原因：LLM验证已覆盖，关键词过滤不灵活且对非标准诊断名有误伤

### 2. 搜索只做一次
- loadDiagnosisKb → smartSearch(diag.name) → 存verifiedResults + verifiedPaths
- loadDiagnosisAdvice → 直接用verifiedPaths加载全文，不再调searchAndLoad/searchByPrefix
- 删掉loadDiagnosisAdvice中的searchAndLoad和searchByPrefix调用

### 3. 教材summary补充
问题：教材md没有summary字段，FTS meta里不含病名关键词
方案A：给50个教材md补summary（手动写，最精确）
方案B：索引时，若summary为空，从content前300字符截取作为fallback summary
- 选方案B：零维护成本，且教材前300字符通常是章节概述含关键主题词

### 4. 提取+归纳合并为一次LLM调用
删掉loadDiagnosisSummary，合并进Step 5

### 5. 提取agent增加段落级相关性约束
prompt明确要求排除跨主题段落，区分"推荐意见"vs"知识要点"

### 6. 全文截断改为token预算制
- 总预算: 6000 input token（中文约12000字符）
- 分配: min(4000, 12000/总篇数) 字符/篇
- 8篇→每篇1500字符，4篇→每篇3000字符，2篇→每篇4000字符上限

### 7. query只用diag.name
不拼description分词碎片

### 8. DiagnosisKbData调整
```
class DiagnosisKbData {
  ruleId: string = '';
  searchResults: SearchResult[] = [];
  verifiedPaths: string[] = [];           // 新增
  recommendations: GuidelineRecommendation[] = [];
  summary: string = '';                   // 从loadDiagnosisSummary合并来
  loadingKb: boolean = false;
  loadingAdvice: boolean = false;
  verifyInfo: string = '';
}
```
删除: expanded, refsExpanded, loadingSummary

### 9. GuidelineRecommendation增加type字段
```
type: string = '';  // "推荐意见" | "知识要点"
```
UI展示时区分样式：推荐意见显示等级标签，知识要点不显示
