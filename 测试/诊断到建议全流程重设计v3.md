# 诊断→建议 全流程重设计 v3

## 对v2的修正

### 修正1：验证不可跳过

v2写"≤3条时跳过验证"——这是错误的。

FTS即使只返回3条，也可能混入不相关文章（metadata索引本身就有噪声，且扩展查询会引入更多噪声）。医疗场景下验证是安全网，不可省略。

**修正：验证始终执行，无论结果条数多少。**

### 修正2：64k上下文下的token预算重算

#### 前提
- 目标LLM：本地部署开源模型，64k上下文
- 中文token比例：约1.5字符/token（现代tokenizer对中文效率提升）
- 医疗准确 > 成本，不需要过度压缩

#### 计算

```
64,000 tokens 总上下文
- system prompt        ≈   800 tokens
- 用户指令/患者摘要    ≈   500 tokens
- 输出预留(recommendations+summary) ≈ 3,000 tokens
- 安全余量             ≈ 1,700 tokens
───────────────────────────────────
可用于文章全文的token  ≈ 58,000 tokens
```

58,000 tokens × 1.5字符/token ≈ **87,000字符**

#### 医学文章典型长度

| 类型 | 典型字符数 | 约token数 |
|------|-----------|----------|
| SSC指南单章节 | 8,000-15,000 | 5,000-10,000 |
| NICE路径单页 | 3,000-8,000 | 2,000-5,000 |
| 教材单章 | 5,000-20,000 | 3,000-13,000 |
| 机制/笔记文章 | 2,000-6,000 | 1,300-4,000 |

#### 不同篇数下的可行性

| 总篇数 | 假设平均长度 | 总字符 | 总token | 是否fit 64k |
|--------|-------------|--------|---------|------------|
| 8篇 | 10,000 | 80,000 | 53,000 | ✅ 有余量 |
| 12篇 | 10,000 | 120,000 | 80,000 | ❌ 超出 |
| 10篇 | 8,000 | 80,000 | 53,000 | ✅ 有余量 |
| 12篇 | 7,000 | 84,000 | 56,000 | ✅ 勉强 |

#### 结论

- **10篇以内、单篇不截断**，64k上下文完全够用
- **10-15篇，单篇截断到7,000字符**，也可fit
- **本地LLM对token不敏感，优先保完整，宁多不截**

### 修正3：5+3篇数限制是否够

#### 问题

验证通过的文章可能有几种情况：
- 冷门诊断：验证通过2-3篇
- 常见诊断（重症肺炎、脓毒症、ARDS）：验证通过8-15篇
- 超常见诊断：验证通过15+篇

5+3=8篇上限，对于重症肺炎这种：
- CAP指南、HAP指南、VAP指南、ATS/IDSA指南、中华肺炎指南 → 5篇指南已满
- 但还有重症肺炎专门章节、呼吸支持相关章节 → 挤不进去
- 教材3章：可能涵盖肺炎、机械通气、感染 → 但遗漏ARDS章节、抗生素章节

**5+3太紧了。**

#### 修正方案

不再固定5+3，改为：

```
验证通过的全部文章都加载
如果总篇数 ≤ 10：全部加载，单篇不截断
如果总篇数 11-15：全部加载，单篇截断到 7000字符
如果总篇数 > 15：取前15篇(按FTS排序)，单篇截断到 7000字符
```

但还有一个问题：**验证通过的篇数可能不够**，不是太多而是太少。

原因：FTS广搜只取30条，验证后可能只剩3-5条。如果30条里关于某诊断的文章本就只有5篇，广搜上限太低就漏了。

**修正：FTS广搜上限从30提到50，给验证agent更大的候选池。**

### 修正4：分类留空原则

提取归纳时，如果某个临床类别（病理生理/诊断/监测/治疗）确实没有检索到相关内容：

**宁可留空，不要LLM猜测或从常识推断。**

prompt中明确：
```
如果某类别在提供的参考资料中确实没有相关内容，该类别输出空数组[]。
严禁从医学常识推断或编造该类别的内容。
空类别比编造内容更安全。
```

### 修正5：LLM调用不硬限，可分批

原则：**医疗准确 > 成本 > 速度**

如果验证通过的文章数量多（>10篇），一次性喂入可能超出上下文或导致LLM注意力稀释，可以分批：

```
if 总篇数 ≤ 10:
    一次调用，喂入全部全文
else:
    按相关度排序（FTS score）
    分2批：
    批次1: 前5篇（最相关）→ 提取→ 得recs_1
    批次2: 后续5-10篇 → 提取→ 得recs_2
    合并: 去重合并 recs_1 + recs_2
```

分批的好处：
- 每批LLM聚焦更少量文章，提取质量更高
- 不会因输入太长导致LLM"注意力稀释"忽略后半部分文章
- 本地LLM不心疼token

但这不是常态，绝大多数诊断验证通过≤8篇，一次调用即可。分批只在文章特别多时触发。

## 修正后完整流程

```
患者数据
  │
  ▼
┌─────────────────────────────────────────┐
│ Step 0: 诊断触发                           │
│ ├ 硬规则匹配 → rule诊断         （0 LLM）  │
│ ├ 无硬规则 → LLM兜底建议        （1 LLM）  │
│ └ 用户手动添加                   （0 LLM）  │
│                                            │
│ 产出: diag.name                            │
└─────────────────────────────────────────┘
  │
  ▼ 诊断卡片出现时触发
┌─────────────────────────────────────────┐
│ Step 1: 查询扩展                （1 LLM）  │
│                                            │
│ 输入: diag.name                            │
│ Agent: expandQueryWithLlm                  │
│                                            │
│ 产出: expandedTerms[]                      │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│ Step 2: FTS广搜                 （0 LLM）  │
│                                            │
│ 查询: diag.name + expandedTerms OR拼接     │
│ 取数: ≤50条（大候选池给验证agent充分筛选）│
│                                            │
│ 产出: broadResults[]（≤50条）              │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│ Step 3: 语义验证                （1 LLM）  │
│                                            │
│ Agent: verifyResultsWithLlm                │
│ 输入: diag.name + broadResults的           │
│       [path, title]列表                    │
│ 任务: 逐条判断主题是否关于该诊断           │
│                                            │
│ ★ 始终执行，无论条数多少                   │
│ ★ 1条也验证（防止1条不相关的噪声）         │
│                                            │
│ 产出: verifiedResults[]                    │
│       verifiedPaths[]                      │
│       verifyInfo调试字符串                  │
│                                            │
│ 全部被拒时: 回退FTS原始排序取前10条         │
│ （宁可有噪声也不返回空，由提取agent兜底）  │
└─────────────────────────────────────────┘
  │
  ▼ 诊断卡片展示指南参考列表
  │
  ▼ 用户点击"查看建议"时触发（懒加载）
┌─────────────────────────────────────────┐
│ Step 4: 加载已验证文章全文      （0 LLM）  │
│                                            │
│ 从kb_content表按verifiedPaths加载          │
│                                            │
│ 加载策略:                                  │
│ ├ 总篇数 ≤ 10: 全部加载，不截断           │
│ ├ 总篇数 11-15: 全部加载，截断7000字符/篇  │
│ └ 总篇数 > 15: 取前15篇，截断7000字符/篇   │
│                                            │
│ 不再区分指南≤5/教材≤3，验证通过即加载      │
│                                            │
│ 产出: pages[]                              │
└─────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────┐
│ Step 5: 提取+归纳               （1~2 LLM）│
│                                            │
│ if 总篇数 ≤ 10:                            │
│   一次调用，喂入全部全文                    │
│ else:                                      │
│   分2批调用，每批≤8篇，合并去重             │
│                                            │
│ 关键指令:                                  │
│ 1. 区分文章体裁:                           │
│    - 结构化指南 → 提取"推荐意见"，         │
│      保留原文推荐等级                      │
│    - 自然段落式 → 提取"知识要点"，         │
│      不编造推荐等级                        │
│ 2. 段落级相关性约束:                       │
│    只提取与{diag.name}主题直接相关的内容    │
│    排除仅提及但非核心的段落                 │
│ 3. 分类留空原则:                           │
│    某类别无相关内容 → 输出空数组[]          │
│    严禁从常识推断编造                      │
│ 4. 同时生成临床要点摘要                    │
│                                            │
│ 输出JSON:                                  │
│ {                                          │
│   "recommendations": [{                    │
│     "text": "...",                         │
│     "evidenceLevel": "1A|2B|未标注",       │
│     "source": "SSC 2021|教材12章",         │
│     "category": "病理生理|诊断|监测|治疗",  │
│     "type": "推荐意见|知识要点"             │
│   }],                                      │
│   "summary": "临床要点摘要"                │
│ }                                          │
│                                            │
│ 产出: recommendations[] + summary          │
└─────────────────────────────────────────┘
```

## LLM调用统计

| 场景 | 调用次数 |
|------|---------|
| 硬规则命中 + ≤10篇验证文章 | 3次（expand + verify + extract） |
| 硬规则命中 + >10篇验证文章 | 4次（expand + verify + extract×2批） |
| LLM兜底诊断 + ≤10篇 | 4次（suggest + expand + verify + extract） |
| LLM兜底诊断 + >10篇 | 5次（suggest + expand + verify + extract×2批） |

典型场景3次，极端场景5次。本地LLM不心疼。

### KnowledgeService.ets

2. **删掉filterByCoreTerms + extractCoreTerms**：LLM验证替代
3. **smartSearch**：
   - 去掉filterByCoreTerms调用
   - FTS广搜上限 30→50
   - 验证始终执行（删掉≤3跳过逻辑）
4. **verifyResultsWithLlm**：删掉≤3条跳过逻辑
5. **删掉searchAndLoad**（不再需要，用verifiedPaths+getPage替代）

### PatientDetailPage.ets

7. **DiagnosisKbData**：增加verifiedPaths，删掉expanded/refsExpanded/loadingSummary
8. **loadDiagnosisKb**：
   - query只用diag.name
   - smartSearch结果存verifiedPaths
   - 删掉verifyInfo的"基础搜索"对比，改为"广搜→验证"
9. **loadDiagnosisAdvice**：
   - 不再调searchAndLoad/searchByPrefix
   - 用verifiedPaths + getPage加载全文
   - 全文截断：≤10篇不截，11-15篇截7000，>15篇取前15截7000
   - 提取+归纳合并为一次LLM调用
   - prompt重写（体裁区分+段落约束+留空原则+type字段）
10. **删掉loadDiagnosisSummary**
11. **GuidelineRecommendation**：增加type字段
12. **UI**：推荐意见显示等级标签，知识要点不显示

### KB数据

13. **KB_DB_VERSION** +1，触发删库重建（summary fallback生效）
14. **教材50章**：无需手动补summary，fallback自动生效

### verifyResultsWithLlm回退逻辑

15. 全部被拒时：回退FTS原始排序取前10条
    - v2是回退ftsResults.slice(0, limit)，limit=10
    - 理由：验证agent可能误拒（LLM判断有假阴性），宁可带噪声进提取agent，由提取agent的段落级约束二次过滤
    - 空结果对临床决策更危险
