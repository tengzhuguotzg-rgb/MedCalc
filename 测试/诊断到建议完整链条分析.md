# 诊断到建议完整链条分析

## 阶段1：诊断触发（无LLM）

| 来源 | 机制 | source字段 |
|------|------|-----------|
| 硬规则引擎 | `DiagnosisRules.ets` 52条纯数据匹配 | `rule` |
| LLM兜底agent | `suggestDiagnosesWithLlm`：硬规则未覆盖时，LLM分析患者数据 | `suggestion` |
| 用户手动 | 用户自行添加 | `user` |

→ 产出：诊断卡片，每个有 `diag.name`（如"ARDS 重度"）+ `rule.description`

---

## 阶段2：KB搜索（loadDiagnosisKb，诊断卡片出现时触发）

### 查询词构造
`diag.name` + `rule.description` 的分词片段，拼成空格分隔（最多4个词）

- 例：`ARDS 重度` → query = `"ARDS 重度 Berlin P/F<100"`

### 搜索流程
1. **扩展agent** (`expandQueryWithLlm`)：将query扩展为更多同义术语
2. **FTS广搜**：扩展后OR搜索，取 `limit*3` 条
3. **filterByCoreTerms**：用原始查询核心词过滤，放宽到 `limit*2`
4. **验证agent** (`verifyResultsWithLlm`)：二值判断每条是否主题相关

→ 产出：`searchResults`（诊断卡片下方"指南参考"区域展示的条目列表）

---

## 阶段3：指南建议提取（点进详情页时触发，loadDiagnosisAdvice）

### 又一次独立搜索
- `searchAndLoad(diag.name, 5)` → 内部又调 `smartSearch(diag.name)`
- `searchByPrefix(diag.name, '实用重症医学/', 3)` → 内部又调 `smartSearch(diag.name)`

→ 产出：guidelinePages + textbookPages（全文内容）

### 提取agent (LLM)
把全文喂给LLM，从参考资料中提取4分类推荐意见（病理生理/诊断/监测/治疗）

→ 产出：`recommendations[]`（详情页展示的临床建议）

---

## 阶段4：临床要点归纳（loadDiagnosisSummary，建议提取后触发）

把 `recommendations` 文本化后喂LLM归纳

→ 产出：`summary`（详情页顶部"临床要点"区域）

---

## 发现的问题：阶段2和阶段3是两套独立搜索

| 维度 | 阶段2（指南参考列表） | 阶段3（建议提取全文源） |
|------|----------------------|------------------------|
| query | `diag.name + description分词` | 仅 `diag.name` |
| 扩展 | expandQueryWithLlm | smartSearch内部又调一次expandQueryWithLlm |
| 验证 | verifyResultsWithLlm | smartSearch内部又调一次verifyResultsWithLlm |
| 结果用途 | 卡片下显示条目列表 | 喂LLM提取建议 |

### "重症肺炎"问题复盘

1. 阶段2搜的是 `"重症肺炎 + description词"`，抓到HAP相关
2. 阶段3搜的是 `"重症肺炎"`，LLM扩展可能带入镇静镇痛相关内容（重症肺炎常涉及镇静）
3. 全文喂给提取agent → 全文里混入跨主题内容（肺炎文章里提到镇静→全文包含PADIS段落）
4. 提取agent忠实提取了这些段落 → 出现与肺炎无关的PADIS指南建议

### 核心矛盾

验证agent只验证**文章级**相关性，不验证**段落级**相关性。

重症肺炎文章全文可能包含：
- 肺炎核心内容（病理生理、抗生素、呼吸支持）✅ 相关
- 镇静镇痛段落（ICU常规提及）❌ 不相关
- 谵妄评估段落（重症患者常规评估）❌ 不相关

提取agent从全文中提取时，无法区分哪些段落是"关于肺炎的"vs"仅提及的"。

---

## 可能的修复方向

### 方向A：阶段3复用阶段2的搜索结果（避免二次搜索）
- 阶段2已做过smartSearch+验证，结果已存在 `kbData.searchResults`
- 阶段3直接用这些已验证的path加载全文，不再独立搜索
- 优点：搜索结果一致，避免重复LLM调用
- 缺点：query不同（阶段2含description词），阶段3只用diag.name可能遗漏

### 方向B：提取agent增加段落级相关性约束
- 在提取agent的systemPrompt中强调：只提取与诊断主题直接相关的建议，忽略仅提及的段落
- 优点：不改搜索逻辑，只收紧提取
- 缺点：依赖LLM判断力，全文太长时可能仍会混入

### 方向C：全文喂入前做段落级过滤
- 加一个"段落级验证agent"：把全文分段，逐段判断是否与查询主题相关，不相关的段落在喂入提取agent前就截掉
- 优点：最精确
- 缺点：多一次LLM调用，全文可能很长，token消耗大

### 方向D：阶段3搜索统一query，搜索结果共享
- 阶段2和阶段3用同一个query（`diag.name + description分词`）
- 阶段2的结果存入kbData，阶段3直接复用
- 优点：逻辑统一，减少重复
- 缺点：description分词对教材搜索可能太窄

### 方向E：截断策略——只喂文章前N字符
- 大部分医学科普/指南文章，核心内容在前半部分，后半部分是鉴别/延伸
- 截断到2000字符而非4000字符，减少跨主题段落混入概率
- 优点：简单
- 缺点：可能丢失文章后半部分的核心治疗建议
